# commons & sense デジタル版 — 企画・設計プラン

> ハイクオリティなファッション/アート/ライフスタイル誌「commons & sense」のデジタル版を、
> VUAL基盤の上に「体験型ライフスタイルEC × エディトリアルマガジン」として構築する提案デモ。

最終更新: 2026-06-04

---

## 0. 前提・スタンス

- **実案件ベース**: 現在 c&s からクリエイティブの仕事を受けている関係。ロゴ・世界観を使用してデモ制作OK。
- **運用主体は自分（Sachio）**: Vercel / サーバー / コードは自分が管理し続ける前提。
- **デモ → 提案 → 契約後に本番ドメインへ移行**という流れ。
- VUAL の既存資産（試着エンジン・Gemini生成・EC/Stripe・マルチテナント・admin）を最大限転用する。

## 1. ホスティング & ドメイン戦略

| フェーズ | URL | 内容 |
|---|---|---|
| デモ | `commons.vual.jp` | VUALマルチテナントのサブドメインとして相乗り |
| 本番（契約後）| `www.commons-sense.net` | 同じVercelプロジェクトに独自ドメインを追加。先方はDNSを向けるだけ |

### 移行の容易さ
- Vercel の Domains に `www.commons-sense.net` を追加（自分の作業・5分）
- 先方は CNAME（`www` → `cname.vercel-dns.com`）/ A レコードを設定するだけ
- SSL は Vercel 自動発行、コード・デプロイ変更不要、ダウンタイムほぼ0
- **ドメイン自体は先方資産**なので、DNS設定は先方に依頼 or 委任が必要

### 設計上の備え（重要）
- 現状の `lib/store-resolver.ts` は `xxx.vual.jp` サブドメイン前提。
- **カスタムドメイン → テナント のマッピング**を最初から見据えて設計する
  （`commons.vual.jp` と `www.commons-sense.net` の両方が同じ commons テナントを指せるように）。
- → 移行時にコードをいじらなくて済む。営業材料にもなる（「契約後すぐ御社ドメインで公開可能」）。

## 2. 情報設計（IA）— 4階層

```
フィード（タブ: All / Fashion / Art / Lifestyle / Culture / Interview、新着順）
  └─ 記事 …… ブロックを縦に積んだ没入型エディトリアル（縦スクロールで読む単位）
       └─ ブロック …… レイアウト固定の器（テンプレ選択型 = B案）
            └─ ホットスポット …… 画像内の試着・購入タグ
```

- **フィード（一覧）** = 記事の入口を並べる層。タブで絞る。
- **記事（詳細）** = ブロックで組まれた没入型の読み物。

---

## 3. フィード画面

### タブ（カテゴリフィルタ）
`すべて(All)` / `Fashion` / `Art` / `Lifestyle` / `Culture` / `Interview`

- デフォルト = **「すべて」**: 全カテゴリを**公開日の新しい順**でどんどん流す（最新号が常に更新される感覚）
- タブタップで該当カテゴリのみに絞り込み
- 無限スクロール

### 記事カードの見せ方 = **A案（大きく没入）**
- キービジュアル大きめ + タイトル + カテゴリラベル
- 1画面に1〜1.5記事。雑誌感・世界観で引き込む

### 将来の余地（今は実装しない）
- 「ピン留め / 特集」フラグ: 編集チームが選んだ記事をトップ固定（新着順だと名記事が埋もれる対策）

## 4. 記事の中身 — ブロック型エディトリアル（B案）

「レイアウトが固定された器（ブロック）」を縦に積んで1記事を構成。
デザイン（余白・フォント・比率・配置）はコードで作り込み固定 → 誰が編集しても c&s クオリティが崩れない。

### ブロックのデータ構造（イメージ）
```
記事 = {
  category: "Fashion",
  publishedAt: ...,
  blocks: [
    { type: "fullBleed",  image, heading, sub },
    { type: "textImage",  side: "right", body, image },
    { type: "gallery",    images: [...] },
    { type: "productRow", products: [productId, ...] },
    { type: "quote",      text },
    { type: "interview",  qa: [{q, a}, ...] },
  ]
}
```

### 初期ブロックテンプレ（雑誌系の定番）— まず6種、後から増やせる設計
1. **全面ビジュアル**（表紙・章扉）
2. **左右分割**（テキスト + 写真）
3. **写真ギャラリー**（複数画像グリッド）
4. **商品特集**（試着付き商品の横並び）
5. **引用 / 大きな一文**（エディトリアルの“間”）
6. **インタビュー**（Q&A形式）

> フロントは `type` を見て対応ブロックコンポーネントを描画するだけ。
> 商品/試着は VUALの既存コンポーネントを `productRow` 等の中で呼ぶ。

### 拡張可能な設計（重要）
- ブロックは **`type` → コンポーネント のレジストリ方式**で管理。
  新テンプレを足す = レジストリに1エントリ追加 + コンポーネント1つ作るだけ。
- 既存6種の**レイアウト修正**もコンポーネント単位で完結（他に影響させない）。
- 不足が分かったら随時追加・修正していく前提。最初の6種は固定確定ではなく出発点。

## 5. ショッパブル画像（試着 + 購入モーダル）

エディトリアル画像に商品ホットスポット（タグ）を埋め込み、世界観を壊さず「読んで→試着→購入」を一気通貫。

### データ
```
{
  type: "fullBleed",
  image: "editorial-01.jpg",
  hotspots: [
    { x: 42, y: 30, productId: "wool-coat" },   // 画像上の%座標 + 商品ID
    { x: 68, y: 55, productId: "leather-bag" }
  ]
}
```

### 体験フロー
1. 記事を読む（縦スクロール）
2. 画像のドットをタップ → 商品モーダル/ボトムシートがせり上がる
3. **「試着」** → VUAL Virtual Try-On 起動（服・アイウェア対応）
4. **「カートに入れる」→「購入」** → Stripe決済
5. モーダルを閉じれば記事に戻る（離脱させない）

### ホットスポットの見せ方 = **B寄り（控えめ表示）**
- 普段は世界観優先で控えめ、ホバー/タップで現れる。雑誌の美しさを邪魔しない。

### 編集チームの操作
- 画像をアップ → 画像上をクリックして「ここに商品タグ」→ 登録済み商品から選択
- Instagram のショッピングタグと同じUX。コード不要。

## 6. 試着・置き試し（AI合成）

| 商材 | 機能 | 実現性 | 時期 |
|---|---|---|---|
| 服 | Virtual Try-On | ◎ 既存エンジン | **Phase 1から** |
| アイウェア | Try-On | ◎ | **Phase 1から** |
| アート/インテリア小物 | **置き試し**（部屋写真に作品を配置合成）| Gemini次第 | **後回し**（技術が追いついたら） |

- 置き試しは試着と同構造（ユーザー写真＋商品 → 合成）。プロンプトと参照画像の差し替え。
- 難所: 遠近・スケール、光と影の整合、遮蔽。最初は「壁掛けアート」「棚/床置き小物」など手前配置から。
- **方針**: 置き試しはできなければ無理せず、Geminiの精度が上がってから。服・アイウェアで先行して価値を出す。

## 8. 記事作成ツール（AIアシスト）— 編集スタッフ自走の核

`vual-x-tool/threads_generator.py` のロジックを転用し、管理パネルのブロックモジュール近くに配置。

### threads_generator.py から流用するロジック
- Anthropic API パイプライン: **本文生成（Sonnet）→ リファイン（Sonnet）**
  - ⚠️ **フック生成は不要**（あれはThreads/SNS用。記事には使わない）
- **ペルソナ注入**（下記）/ トーン
- **Web検索リサーチ結果を生成に差し込む**仕組み（確信が持てない事実は断定しない安全装置あり）

### ペルソナ（投稿者が選択）
記事作成時に投稿者＝執筆スタッフを選ぶと、そのペルソナの文体で生成される。

| ペルソナ | 設定（初期値・後から調整可）|
|---|---|
| **USHIJIMA** | 辛口・40歳。歯に衣着せない、批評的で芯のある文体 |
| **KIMOTO** | 柔らかい口調・25歳。親しみやすく丁寧、寄り添う文体 |

- threads_generator の `persona_id` 方式を踏襲（ペルソナごとに system_prompt を持つ）。
- 初期値は上記。**後から口調・年齢・人物を追加/調整できる**設計にする。

### 記事作成フロー（目標: 8割自動）
```
① 投稿者（USHIJIMA / KIMOTO）を選択
② 入力: 最新ニュースのURLを貼る or キーワードを入れる
③ リサーチが走る（URL本文取得 / Web検索）
④ AIが記事ドラフトを生成（選択ペルソナの文体 + ブロック構成案つき = 8割完成）
⑤ スタッフが手直し（ブロック編集・画像差し替え・トーン調整）
⑥ パブリッシュ → フィード先頭に反映
```

### UI要件
- ブロックモジュールの**近く**に記事作成ツールを配置（執筆と構成がシームレス）
- **画像ドラッグ&ドロップ**対応（アップロード・ブロックへの配置）
- 生成結果はブロック配列として出力 → そのまま編集可能な状態でエディタに展開

### 注意
- `threads_generator.py` は Threads投稿用（短文・SNS口調）。c&s記事用に**プロンプトを雑誌エディトリアル調に作り直す**必要あり。
- **フック生成ロジックは移植しない**（SNS専用機能）。記事は本文生成→リファインのみ。
- ペルソナは USHIJIMA / KIMOTO の2名から作り直し（threads_generator の `persona_id` 構造を流用）。
- 確信の持てない事実（素材・価格・スペック・数字）は断定しない安全装置は流用。

## 9. 管理パネル（編集チーム自走）

「骨組み（レイアウト・世界観）は固定 / 中身（記事・商品）は編集自由」の分離設計。

| やりたいこと | VUAL既存機能 | 流用度 |
|---|---|---|
| 記事の追加/編集 → フィード更新 | `admin/blog` + `(customer)/blog` | ◎ |
| 商品出品 → 商品セクション更新 | `admin/products` | ◎ |
| コレクション/特集編集 | `admin/collection` | ◎ |
| 在庫・価格・公開/非公開 | products管理 | ◎ |

- **権限**: マルチテナント + ロールで、c&s編集チームは c&sテナントのコンテンツのみ編集可。レイアウト/設定/他テナントは触れない。
- 記事公開 → 自動でフィード先頭（新着順）に反映。並び順を気にしなくていい。

## 10. 決済（Stripe）

- **デモ**: 既存アカウントの**テストモード**キー（`sk_test_` / `pk_test_` / `whsec_`）を**デモ用デプロイのenvにだけ**設定。本番VUALの決済に影響なし。
- テストカード `4242 4242 4242 4242` で決済完了画面・在庫減算まで演出可能。
- **本番化**: 契約後に `sk_live_` へenv差し替えるだけ。
- VUALには Stripe決済が実装済み（`app/api/billing/*`, `app/api/checkout`, webhook）。

## 12. ロードマップ（提案）

### Phase 0 — 設計確定（このドキュメント）
- IA・ブロック種別・カテゴリ・UI方針を固める ← **今ここ**

### Phase 1 — 土台（commons.vual.jp テナント）
